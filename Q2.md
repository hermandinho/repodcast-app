## Framing — why acquisition, not features

- Q1's original NextMove.md Q2 leaned product (post-publish analytics,
  Slack, CMS push, style guide, newsletter push). Two of those — the
  analytics loopback (#2) and voice drift (#18, deferred to Q4 in the
  original plan) — are **stillborn without customer data**. You cannot
  tune a "which voice variants perform" loop when no post has ever
  been published through the platform.
- The product surface is already broader than most competitors on
  paper (7 posts + clips + artwork + audiograms + per-client voice
  engine). More features without users is worse than fewer features
  with 25 paying agencies.
- The pieces we DO need before scale — signup attribution, funnel
  instrumentation, a public sample of the output, a launchable
  demo — are missing today and cheap to build.

**Q2 target.** 25 paying agencies on any tier by end of week 26,
either SOLO/STUDIO trial-converts or Agency/Network direct
subscriptions. That's ~2 signups per week post-launch. Read this
number as the smoke test for whether the current pitch works — not a
revenue target.

## Weeks 14–16 — Distribution infrastructure

Everything in this phase is a prerequisite for weeks 17+ to be
data-driven. Don't skip.

### 1. Signup attribution — new `AgencyAttribution` model (week 14)

Prisma migration:

```prisma
model AgencyAttribution {
  agencyId       String   @id
  agency         Agency   @relation(fields: [agencyId], references: [id], onDelete: Cascade)
  utmSource      String?  // e.g. "producthunt", "twitter", "linkedin", "cold-email"
  utmMedium      String?  // e.g. "referral", "social", "email"
  utmCampaign    String?  // e.g. "launch-day", "outreach-w17"
  utmContent     String?
  utmTerm        String?
  referrer       String?  // document.referrer at first landing
  landingPath    String?  // pathname of the first page visited
  signupPath     String?  // pathname where /sign-up was hit
  gclid          String?  // Google Click ID (future paid)
  fbclid         String?  // Facebook Click ID (future paid)
  createdAt      DateTime @default(now())
}
```

Why a sidecar table and not columns on `Agency`:

- Attribution is write-once, read-rarely. Keeping it off the
  hot `Agency` row keeps that row lean.
- Q3 may add per-touchpoint attribution (multi-touch) — a sidecar
  extends more cleanly to that shape.

Client-side capture in `app/layout.tsx` or a dedicated
`components/analytics/attribution-capture.tsx` client component:

- On first landing, read `document.referrer` +
  `new URL(location.href).searchParams` and stash both in a
  `localStorage['repodcast.attr']` blob + a first-party cookie
  (`repodcast_attr`, `Max-Age=2592000`, `SameSite=Lax`, HTTP-only
  false so client can read on `/sign-up`).
- On `createAgencyAction` success, read the cookie/blob, upsert an
  `AgencyAttribution` row. Purge the cookie post-write.

Server-action: `server/db/attribution.ts` exposes `captureAttribution`

- `getAttributionFor(agencyId)`. Root dashboard surfaces the funnel
  by source in Q3.

### 2. PostHog funnels + dashboards (week 14)

Event taxonomy already exists (`lib/analytics/track-client.ts`).
Verify the following are firing and add if missing:

- `landing_hero_viewed`
- `pricing_viewed`
- `signup_started` (Clerk widget mounted)
- `signup_completed` (Clerk `user.created` webhook)
- `onboarding_step_completed` (per step of the 3-step wizard)
- `first_episode_created`
- `first_generation_completed`
- `first_output_approved`
- `first_clip_generated` (Q1 feature, no existing event)
- `trial_activated` (first successful checkout)
- `trial_converted` (first non-trial invoice paid)

Build the funnel in PostHog:

```
landing_hero_viewed
  → pricing_viewed
  → signup_started
  → signup_completed
  → onboarding_step_completed (step=3)
  → first_generation_completed
  → first_output_approved
  → trial_activated
  → trial_converted
```

The interesting numbers are the between-step drop rates. Whichever
step drops the most is what week 17 fixes.

Ship a `/root/funnels` server-rendered mirror of the PostHog funnel so
we're not tied to PostHog UI availability. Queries hit PostHog HogQL
API server-side; PostHog service token in env, not client-side.

Not a redesign. A **five-person usability test** using existing

## Weeks 17–19 — Trial-to-paid funnel polish

Now we have data. The PostHog funnel from week 14 is showing us where
people drop.

### Instrumentation-driven fixes (weeks 17–18)

For each of the top three drop-offs (whatever they turn out to be —
DO NOT guess in advance), ship one fix. Common candidates and their
likely fixes:

- **Drop at `signup_completed → onboarding_step_completed step=1`**:
  the wizard's first-touch is confusing. Fixes: shorter copy, a
  single-field workspace-name skeleton, better empty-state art.
- **Drop at `onboarding_step_completed step=3 → first_episode_created`**:
  users don't know what to do after onboarding. Fix: a persistent
  post-onboarding "Try a sample episode" nudge on the dashboard that
  seeds a real episode from a bundled MP3.
- **Drop at `first_generation_completed → first_output_approved`**:
  users don't know approval is a step, or the approval UI is
  ambiguous. Fix: a coach-mark on the first output card ("Approve to
  train the voice engine").
- **Drop at `first_output_approved → trial_activated`**: value
  discovery happens too slowly to warrant the $1 activation. Fix:
  surface trial framing earlier (in-app banner: "3 more days before
  trial ends"), and ensure the "first launch kit" moment (below)
  actually lands.

Constraint: **one fix per drop-off, one drop-off per week.** Resist
scope creep. Ship, redeploy, watch the number move.

### "First launch kit" reward moment (week 19)

The first time a user completes the four pipelines on one episode
(text + clips + artwork + audiograms all ≥1 output ready), fire a
one-time celebration:

- Modal that stacks the four deliverable types visually, plus a "share
  this delivery" tokenized link (reuses the existing client-portal
  link machinery).
- Persist a `MemberAchievement` row so it only fires once per member.
- PostHog event `first_launch_kit_completed` — the graduation
  marker for whether the value moment has landed pre-trial-conversion.
- Copy: "That's one launch kit. In your client's voice. In 4 minutes.
  Do this for every episode this month." Then a link to `/episodes/new`.

Data model:

```prisma
model MemberAchievement {
  memberId    String
  member      Member  @relation(fields: [memberId], references: [id], onDelete: Cascade)
  kind        MemberAchievementKind
  achievedAt  DateTime @default(now())

  @@id([memberId, kind])
}

enum MemberAchievementKind {
  FIRST_LAUNCH_KIT
  FIRST_APPROVAL
  FIRST_CLIENT_INVITE
  FIRST_PORTAL_SHARE
}
```

Only `FIRST_LAUNCH_KIT` fires a UI reward in Q2; the others exist as
future hooks and PostHog markers.

## Weeks 21–23 — Style guide PDF + Slack (flagship closer feature)

### Style guide PDF ingest (#12) — week 21

Full spec in `NextMove.md` §12. Q2 concretization:

**Data model** (Prisma migration — week 21):

```prisma
model ShowStyleGuide {
  id             String   @id @default(cuid())
  showId         String   @unique
  show           Show     @relation(fields: [showId], references: [id], onDelete: Cascade)
  sourcePdfUrl   String   // R2 key
  sourceFilename String
  extractedAt    DateTime @default(now())
  rulesJson      Json     // { tonalRules: string[], forbiddenWords: string[],
                          //   preferredPhrases: string[],
                          //   perPlatformOverrides: { LINKEDIN: {...}, ... } }
  supersededAt   DateTime?
  agencyId       String   // denormalized for tenant filters
}
```

**Pipeline** (`server/ai/style-guide.ts`):

- Accept a PDF upload (2-step R2 flow — same shape as
  `signSourceVideoUploadAction` from Q1).
- On `finalizeStyleGuideUpload`, dispatch `show/styleGuide.requested`
  Inngest event.
- Inngest fn `extract-style-guide.ts`:
  1. Fetch PDF from R2.
  2. Send to Anthropic API as a `document` content block.
  3. Prompt with `input_schema` tool for `{ tonalRules, forbiddenWords,
preferredPhrases, perPlatformOverrides }`.
  4. Persist `ShowStyleGuide` row.
  5. Copy `preferredPhrases` into `Show.globalInstructions` (append,
     not replace — existing instructions stay).
  6. Insert per-platform overrides into `ShowPlatformInstruction`.

**UI** (weeks 21–22):

- New tab on `/dashboard/shows/[showKey]` → `Style guide`.
- Upload dropzone → preview of extracted rules → "Apply to show"
  button.
- Read-only view of the applied rules with a "Re-extract from source"
  button (useful when the client updates their brand doc).

**Test coverage:**

- Unit test on the extraction prompt with 3 fixture PDFs (short, long,
  scan-based).
- Tenant-isolation test: `ShowStyleGuide` must not leak across
  agencies.

**Ship gate.** All plans — this is an onboarding-friction remover,
not a tier feature.

## Risks and open questions

- **PostHog funnel data is thin.** With <200 signups in weeks 14–17,
  the "top three drop-offs" identification is noisy. Mitigation:
  don't fix drop-offs that aren't at least 2× the baseline rate
  across the funnel. If everything looks flat, ship the "first
  launch kit" moment anyway (it's high-conviction independent of
  data).
- **Launch-week signups exceed capacity.** Q1's VPS is one node. If
  the launch drives >50 clip generations in a burst, queue depth
  will grow. Mitigation: if the queue exceeds 100 pending jobs at
  any point, provision a second CPX11 (€4.35/mo) as a peaker. Same
  bootstrap script.
- **The public sample page cannibalizes signups.** If cold traffic
  gets what they need from `/samples/[slug]` without signing up, we
  lose the conversion. Mitigation: the sample shows OUTPUTS, not the
  voice engine — the voice-fidelity claim only lands when they see
  it work on THEIR content. A/B the CTA aggressiveness in week 20
  after we have baseline conversion data.
- **WordPress integration attack surface.** Application Passwords
  are basic-auth over HTTPS. Store the encrypted password at rest
  via the same envelope encryption Buffer credentials already use
  (`server/security/envelope.ts`). Do NOT log the plaintext even in
  Sentry.
- **Beehiiv rate limits.** v2 API is 60 req/min per publication.
  Fine for our volume; document as a known ceiling in
  `docs/integrations.md` (future).
- **Style guide PDF quality varies wildly.** A brand doc might be
  10 pages of colour palettes and one paragraph of voice guidance.
  Extraction accuracy will be lower on visually-heavy PDFs.
  Mitigation: surface the extracted JSON in the UI before "Apply,"
  let the operator edit before applying.
- **Slack webhook URL leakage.** Webhook URLs are secrets. Encrypt
  at rest, never expose in the UI after save (mask to
  `https://hooks.slack.com/services/T•••/B•••/•••`).
